# 质量 100% 路线图

> **现实**：没有 CI 能 100% 保证零缺陷，但可以无限接近。本文档列出分阶段改进方案。

---

## 一、已实现（当前 CI）

| 检查项 | 说明 |
|--------|------|
| ESLint（含 --max-warnings 0） | 任何 warning 也会导致失败 |
| 数据通道规范 | API/WS 使用符合规范 |
| 重复代码 (jscpd) | 超过 50% 重复则失败 |
| 依赖循环 (depcruise) | 循环依赖、孤立模块 |
| **前端 ESLint** | 游戏 + GM 客户端代码规范 |
| 构建 + 单元测试 + 覆盖率 | 类型错误、逻辑错误 |
| 集成测试 | GM API、玩家 API、防漏洞、WebSocket |
| **E2E 测试** | Playwright 覆盖登录、注册、创建角色 |
| **pre-commit** | 提交前自动 lint + test（husky） |

---

## 二、待实现（分阶段）

### 阶段 1：提高覆盖率阈值（中等工作量）

**现状**：覆盖率阈值 7%，已纳入 item-type、input-sanitize、password、memory、skill、state-machine、errorHandler、shop.getCurrencyMap 等单元测试。

**目标**：逐步提高到 20% → 40% → 60%。

**步骤**：
1. 为核心 service（bag、battle、shop、trade）补充单元测试
2. 每完成一批，在 `jest.config.js` 中提高 `coverageThreshold`
3. 新代码合并前需保证覆盖率不降

---

### 阶段 2：前端 Lint（需改造）

**现状**：客户端 JS 无 lint，全局变量多、script 顺序加载。

**目标**：前端也能跑 ESLint。

**步骤**：
1. 在 `eslint.client.config.js` 中声明所有全局变量（State、API、Pages、navigateTo 等）
2. GM 使用 ES Module，游戏端使用 script，需分别配置 `sourceType`
3. 或：将游戏前端迁移到 ES Module + 打包工具，再统一 lint

---

### 阶段 3：E2E 测试（高工作量）

**目标**：用 Playwright/Cypress 覆盖关键流程。

**建议覆盖**：
- 注册 → 登录 → 创建角色
- 背包：使用物品、穿戴装备、删除
- 战斗：开始、自动战斗、停止
- 商店：购买
- 拍卖：上架、购买

**步骤**：
1. 安装 Playwright
2. 编写 `e2e/login.spec.js` 等
3. CI 中增加 `e2e` job（需启动服务 + 浏览器）

---

### 阶段 4：提交前钩子（可选）

**目标**：本地 commit 前自动跑 lint + 测试。

```bash
# .husky/pre-commit
cd text-rpg-game/server && npm run lint && npm run test
```

需安装 husky + lint-staged。

---

## 三、优先级建议

| 优先级 | 项目 | 工作量 | 收益 |
|--------|------|--------|------|
| 高 | 阶段 1：提高覆盖率 | 中 | 高 |
| 中 | 阶段 2：前端 Lint | 中 | 中 |
| 低 | 阶段 3：E2E | 高 | 高 |
| 低 | 阶段 4：pre-commit | 低 | 中 |

---

## 四、当前 CI 能保证什么

- ✅ 后端编译、类型、规范、单元测试、集成测试、安全测试
- ✅ 重复代码、依赖结构
- ❌ 前端逻辑错误（需阶段 2）
- ❌ 端到端用户流程（需阶段 3）
- ❌ 覆盖率下降（需阶段 1）

**结论**：按路线图推进，可逐步逼近「每次提交都有质量保障」的目标。
