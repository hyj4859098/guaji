## 二、防作弊：为什么「实时战斗」更合适

### 2.1 两种模式对比

| 模式 | 流程 | 防作弊 | 复杂度 |
|------|------|--------|--------|
| **实时战斗** | 客户端点「开始」→ 服务端一口气算完 → WebSocket 推送结果 | ✅ 全部在服务端算，客户端只展示 | 低 |
| **分步战斗** | 客户端每回合点「下一步」→ 服务端算一步 → 返回 | ✅ 也是服务端算，但需要维护「当前回合」状态 | 高 |

**结论**：两种都能防作弊（核心逻辑都在服务端），但**实时战斗更简单**：
- 客户端只发「开始战斗」和「停止战斗」
- 不参与任何伤害、命中、掉落的计算
- 服务端算完直接推送，客户端只负责显示

### 2.2 防作弊原则（必须遵守）

```
✅ 正确：客户端 → "我要打怪物 A，10 场" → 服务端算完 → 推送结果
❌ 错误：客户端 → "我造成了 999 伤害" → 服务端相信并结算
```

**当前设计已经满足**：`executeSingleBattle` 全部在服务端执行，客户端无法篡改结果。

## 三、推荐方案：统一为「实时战斗 + 单一战斗引擎」

### 3.1 整体思路

```
┌─────────────────────────────────────────────────────────────┐
│                      服务端（唯一权威）                        │
│                                                             │
│  ┌─────────────────┐     ┌─────────────────────────────┐  │
│  │ 战斗核心 runBattle │ ←── │ 在线战斗 / 离线战斗 共用同一套  │  │
│  │ (伤害/命中/掉落)   │     │ 公式，只是「是否推送事件」不同  │  │
│  └─────────────────┘     └─────────────────────────────┘  │
│           ↑                          ↑                     │
│           │                          │                     │
│  在线：WebSocket 推送事件    离线：登录时批量结算，不推送     │
└─────────────────────────────────────────────────────────────┘
                              ↑
┌─────────────────────────────────────────────────────────────┐
│                      客户端（只展示）                         │
│  发送：开始战斗、停止战斗、自动喝药配置                        │
│  接收：战斗事件流 → 显示日志、血条、奖励                       │
└─────────────────────────────────────────────────────────────┘
```

### 3.2 具体改动

| 项目 | 做法 | 目的 |
|------|------|------|
| **去掉分步战斗** | 删除 `stepBattleStates`、`nextStep` 相关代码 | 只保留一套逻辑，简单 |
| **保留实时战斗** | 继续用 `executeSingleBattle` + WebSocket | 防作弊、逻辑清晰 |
| **抽取战斗核心** | 把伤害、命中、掉落抽到 `battle-core.ts` | 在线/离线共用，好扩展 |

## 四、离线战斗（后续实现思路）

### 4.1 逻辑（和在线完全一致）

- 用**同一套** `runBattle` 公式
- 区别只在于：在线会推送事件给前端，离线只算结果、不推送

### 4.2 实现步骤（你加离线时参考）

1. **记录离线时间**
   - 用户下线时：存 `last_online_time`
   - 用户上线时：`offline_seconds = now - last_online_time`

2. **计算可战斗场数**
   - 例如：每 30 秒一场 → `max_battles = offline_seconds / 30`

3. **循环执行战斗**
   ```
   for i = 1 to max_battles:
     result = runBattle(uid, enemyId)  // 和在线用同一函数
     if result.失败:
       break  // 失败就停
     else:
       发放奖励  // 经验、金币、随机掉落
   ```

4. **随机掉落**
   - 沿用现有 `processDrop`，每场独立随机，和在线一致

### 4.3 「不能太假」的保证

- 公式、命中、伤害、掉落：和在线完全一样
- 失败立刻停止，胜利才继续
- 每场掉落独立随机，不预生成

## 五、代码结构（简单、好维护）

### 5.1 目标结构

```
server/src/
├── battle/
│   ├── core.ts              # 纯函数：伤害、命中、单回合结算
│   ├── battle.service.ts    # 在线战斗：调用 core + 推送事件
│   └── offline.service.ts   # 离线战斗（后续）：调用 core，不推送
```

### 5.2 core.ts 示例（方便你加新属性、buff）

```typescript
// 命中判定（以后加 buff 就改这里）
export function calculateHit(attacker, defender) { ... }

// 单回合伤害（加新属性就改这里）
export function calcRoundDamage(player, monster) { ... }

// 单场战斗（纯计算，不碰数据库、不推送）
export function runBattle(player, monster): BattleResult { ... }
```

- 新属性：在 `calcRoundDamage` 里加参数
- 新 buff：在 `calculateHit`、`calcRoundDamage` 里加判断

### 5.3 前端保持简单

- 只发：`开始战斗`、`停止战斗`、`自动喝药配置`
- 只收：战斗事件 → 更新日志、血条、奖励
- 不参与任何计算

## 六、分阶段实施建议

### 阶段 1：先简化（1–2 天）

1. 删除分步战斗相关代码（`stepBattleStates`、`nextStep`、对应 API）
2. 前端去掉「分步战斗」入口，只保留「开始战斗」「自动战斗」「停止」
3. 验证：在线战斗行为不变

### 阶段 2：抽核心（1–2 天）

1. 新建 `battle/core.ts`，把 `calculateHit`、伤害公式、`processDrop` 抽进去
2. `executeSingleBattle` 改为调用 `core.runBattle`，自己只负责推送事件、读库
3. 验证：战斗结果和之前一致

### 阶段 3：加离线（你准备做时）

1. 新建 `offline.service.ts`
2. 登录时检查 `last_online_time`，计算 `max_battles`
3. 循环调用 `core.runBattle`，失败即停，胜利发奖励
4. 可选：给用户一个「离线战斗报告」弹窗

## 七、总结

| 你的诉求 | 方案对应 |
|----------|----------|
| 保持当前战斗机制 | 不改公式，只做抽取和整理 |
| 防作弊 | 全部用实时战斗，逻辑在服务端，客户端只展示 |
| 离线战斗 | 用同一套 core，登录时批量结算 |
| 代码简单 | 删分步、统一入口、抽 core |
| 可维护可扩展 | core 里集中改伤害、命中、buff |

**建议执行顺序**：先做阶段 1（删分步），再阶段 2（抽 core），最后在你需要时做阶段 3（离线）。
